## 第6页：数据库设计（40秒）

```
标题：4张云数据库集合设计

  users ────1:N──── goods
    │                 │
    │ 1:N             │ 1:N
    ▼                 ▼
collections        messages

集合详情：
┌────────────┬──────────────────────┬──────────┐
│   集合     │      核心字段          │ 安全策略  │
├────────────┼──────────────────────┼──────────┤
│ users      │ openid, 昵称, 认证状态 │ 仅本人   │
│ goods      │ 标题, 价格, 图片, 分类 │ 公开读   │
│ collections│ 用户+商品关联          │ 仅本人   │
│ messages   │ 发送/接收者, 内容      │ 参与者   │
└────────────┴──────────────────────┴──────────┘

关键设计决策：
- 敏感字段(手机/学号)由云函数抹除后返回 → PII保护
- 所有权校验在云函数层完成 → 前端无法绕过
```

**要点**：突出安全和隐私保护设计

## 第7页：云函数设计（35秒）

```
标题：6个云函数 — BFF模式

getUserInfo（241行 · 11个action合一）
  ├── get / register / update      用户管理
  ├── toggleCollection / getCollections 收藏
  ├── sendMessage / getMessages / getConversations 聊天
  └── getUserById / getUsersByIds  用户查询

其他云函数：getGoodsList · getGoodsDetail
             publishGoods · updateGoods · searchGoods

为什么合并getUserInfo？
✓ 减少冷启动次数（云函数有冷启动延迟）
✓ 同业务逻辑内聚，方便维护
✓ 通过action参数路由，不增加调用复杂度
```

**要点**：解释关键架构决策

## 第13页：遇到的挑战与解决方案（40秒）★ 加分页

```
标题：技术挑战与解决

问题1：微信新登录策略 → getUserProfile返回默认值
解决 → 改用原生 chooseAvatar + type="nickname" 组件

问题2：TabBar页面无法navigateTo传参（编辑功能失效）
解决 → globalData暂存ID → switchTab → onShow加载

问题3：商品卡片价格不显示
解决 → 重构卡片WXML+WXSS，价格使用专用样式类

问题4：云函数部署后 FUNCTION_NOT_FOUND
解决 → app.json补充 "cloud": true

问题5：自己可以给自己发消息（业务逻辑漏洞）
解决 → 云函数层校验 seller_id ≠ 当前用户
```

**要点**：展示问题排查能力和工程素养

## 附录：PPT制作建议

| 项目 | 建议 |
|------|------|
| **模板** | 简洁白色底 + 紫色(#7C3AED)强调色，与小程序本身风格统一 |
| **配图** | 每页至少1张截图或架构图，减少纯文字页 |
| **字体** | 标题用加粗，正文≥24pt（投影仪远距离可读） |
| **动画** | 使用淡入/滑入，不要用弹跳/旋转等花哨动画 |
| **演示** | 微信开发者工具提前打开，切换Tab到需要演示的页面，准备好数据 |
| **备用** | 导出PDF版，防止PPT软件兼容问题 |
