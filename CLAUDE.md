# ============================================
# 项目级 AI 工程规范
# Enterprise AI Engineering Rules
# ============================================

## 1. 项目概述

本项目是一个生产级软件系统。

AI 助手（Claude / GPT / Cursor / Copilot）在参与开发时，
必须严格遵循以下工程规范、架构约束、代码风格与安全规则。

目标：

- 保持代码长期可维护
- 保持架构稳定
- 避免 AI 产生不可控重构
- 保证代码一致性
- 保持工程专业化

AI 的职责：

- 辅助开发
- 遵循现有架构
- 复用已有实现
- 最小化修改范围
- 保持代码稳定

而不是：

- 擅自“优化”
- 大规模重构
- 引入新架构
- 随意修改目录结构
- 重写已有稳定逻辑

--------------------------------------------------

# 2. AI 行为总规则（最高优先级）

## 必须遵守

在生成任何代码前：

1. 先阅读相关文件
2. 理解现有架构
3. 理解已有实现方式
4. 优先复用现有代码
5. 保持风格一致
6. 最小化修改范围
7. 避免影响已有功能
8. 优先保证稳定性

## 禁止行为

AI 禁止：

- 擅自重构项目
- 擅自升级依赖
- 擅自修改数据库结构
- 擅自修改 API 格式
- 擅自修改核心业务逻辑
- 擅自移动文件
- 擅自重命名模块
- 擅自引入新框架
- 创建重复功能
- 写“临时方案”
- 留 TODO 不实现
- 生成伪代码替代真实实现

--------------------------------------------------

# 3. 开发原则

遵循以下工程原则：

- SOLID
- DRY
- KISS
- Separation of Concerns
- Single Responsibility
- High Cohesion
- Low Coupling

必须：

- 保持模块边界清晰
- 保持职责单一
- 保持可测试性
- 保持可维护性

--------------------------------------------------

# 4. 目录结构规范

## 前端结构

```txt
src/
├── api/              # API 请求
├── components/       # 通用组件
├── pages/            # 页面
├── layouts/          # 布局
├── stores/           # 状态管理
├── hooks/            # 自定义 hooks
├── utils/            # 工具函数
├── types/            # TS 类型
├── constants/        # 常量
└── styles/           # 全局样式
```

规则：

- 页面逻辑不得写入 components
- API 请求必须统一管理
- 禁止跨层调用
- utils 不允许包含业务逻辑
- hooks 必须职责单一

--------------------------------------------------

## 后端结构

```txt
backend/
├── controller/
├── service/
├── repository/
├── entity/
├── dto/
├── vo/
├── config/
├── middleware/
├── utils/
└── tests/
```

规则：

- Controller 只负责请求处理
- Service 只负责业务逻辑
- Repository 只负责数据访问
- 禁止跨层操作数据库
- 禁止 Controller 写业务逻辑
- Entity 不允许直接暴露给前端

--------------------------------------------------

# 5. 编码规范

## 命名规范

### 变量

使用：

```txt
camelCase
```

禁止：

```txt
user_name
aaa
tmp
data2
```

--------------------------------------------------

### 类名

使用：

```txt
PascalCase
```

例如：

```txt
UserService
OrderController
AuthMiddleware
```

--------------------------------------------------

### 常量

使用：

```txt
UPPER_CASE
```

--------------------------------------------------

### 文件命名

前端：

```txt
kebab-case
```

后端：

依据语言规范。

--------------------------------------------------

# 6. TypeScript 规范

必须：

- 开启 strict mode
- 禁止 any
- 优先 interface
- 明确返回值类型
- 避免类型断言滥用

禁止：

```ts
const data: any = xxx
```

推荐：

```ts
interface UserInfo {
  id: number
  name: string
}
```

--------------------------------------------------

# 7. API 规范

遵循 RESTful 风格。

## 正确示例

```txt
GET    /users
GET    /users/:id
POST   /users
PUT    /users/:id
DELETE /users/:id
```

## 错误示例

```txt
/getUser
/updateUserInfo
/deleteUserById
```

--------------------------------------------------

## API 返回结构

统一：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

禁止返回格式不统一。

--------------------------------------------------

# 8. 数据库规范

## 表命名

使用：

```txt
snake_case
```

例如：

```txt
user_info
order_record
```

--------------------------------------------------

## 字段规范

所有表必须包含：

```txt
id
created_at
updated_at
```

--------------------------------------------------

## 禁止

- SELECT *
- 无索引查询
- N+1 查询
- 在循环内频繁查询数据库

--------------------------------------------------

# 9. 安全规范

必须：

- 校验所有用户输入
- 使用参数化查询
- 对敏感信息加密
- 使用权限校验
- 校验 JWT
- 防止 XSS
- 防止 CSRF
- 防止 SQL 注入

禁止：

- 硬编码密钥
- 在日志输出密码
- 明文存储敏感信息

--------------------------------------------------

# 10. 日志规范

必须：

- 使用统一日志系统
- 输出错误上下文
- 输出 traceId

禁止：

- console.log 到处输出
- 打印敏感数据
- 无意义日志

--------------------------------------------------

# 11. 错误处理规范

禁止：

```ts
catch (e) {}
```

必须：

- 输出错误日志
- 返回统一错误格式
- 保持错误可追踪

--------------------------------------------------

# 12. 性能规范

必须：

- 减少重复渲染
- 避免重复请求
- 合理缓存
- 懒加载大型模块
- 控制包体积

禁止：

- 无意义 useEffect
- 死循环请求
- 巨型组件
- 巨型函数

--------------------------------------------------

# 13. 函数规范

单函数：

- 不超过 80 行
- 单一职责
- 参数不超过 5 个

复杂逻辑必须拆分。

--------------------------------------------------

# 14. 文件规范

单文件：

- 不超过 500 行

超过后必须拆分。

--------------------------------------------------

# 15. 注释规范

注释必须解释：

- 为什么这样做
- 业务背景
- 边界条件

不要写废话注释。

错误示例：

```ts
// 定义用户变量
const user = {}
```

--------------------------------------------------

# 16. Git 提交规范

格式：

```txt
feat:
fix:
refactor:
docs:
test:
style:
```

例如：

```txt
feat: 添加用户登录功能
fix: 修复 JWT 过期问题
```

--------------------------------------------------

# 17. 测试规范

提交前必须：

- lint 通过
- build 成功
- 单元测试通过

新增业务逻辑必须包含测试。

--------------------------------------------------

# 18. UI 规范

必须：

- 保持统一设计风格
- 控制颜色数量
- 保持间距统一
- 响应式布局
- 适配移动端

禁止：

- 随意使用颜色
- 弹窗套弹窗
- 内联样式泛滥

--------------------------------------------------

# 19. AI 输出规范（极重要）

AI 生成代码时：

必须：

1. 先分析已有代码
2. 优先复用现有模块
3. 保持风格一致
4. 最小化改动
5. 输出完整可运行代码
6. 不生成伪代码
7. 不省略关键实现

--------------------------------------------------

## AI 修改代码时

必须：

- 说明修改原因
- 说明影响范围
- 保持向后兼容
- 避免破坏现有逻辑

--------------------------------------------------

## AI 不确定时

禁止猜测。

必须：

- 明确说明不确定点
- 请求更多上下文

--------------------------------------------------

# 20. 禁止的 AI 行为（最终规则）

AI 严禁：

- 自作聪明“优化”
- 大规模重构
- 修改核心架构
- 偷换技术栈
- 引入不必要复杂度
- 创建重复代码
- 忽略现有实现
- 擅自升级依赖
- 编造不存在 API
- 编造不存在库
- 输出无法运行代码

--------------------------------------------------

# 21. 最终目标

AI 的核心目标：

像一个：

- 稳定
- 谨慎
- 专业
- 高级
- 可协作

的资深工程师。

而不是：

一个会自动生成代码的随机函数。
